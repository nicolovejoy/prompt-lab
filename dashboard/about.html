The Ralph Wiggum Technique

Back to Dashboard

What Is It?

A bash-based methodology for autonomous AI coding loops that breaks work into three phases. Each iteration processes one task with fresh context to maintain efficiency.

"Context is scarce... Plans are disposable... Backpressure beats direction"

Why It Works

  • Token scarcity: With ~176K usable tokens from a 200K window, bloated prompts waste resources
  • Plan drift: Regenerating stale plans is cheaper than salvaging them
  • Backpressure over direction: Engineering conditions where wrong outputs get automatically rejected beats explicit instructions

Three-Phase Workflow

  1. Requirements: Human-LLM dialogue producing spec files in specs/
  2. Planning: Agent generates IMPLEMENTATION_PLAN.md without writing code
  3. Building: One task per iteration; agent implements, validates, updates plan, exits

Five Essential Files

  • loop.sh - bash orchestrator managing iterations
  • PROMPT_plan.md / PROMPT_build.md - mode-specific instructions
  • AGENTS.md - build/test/lint commands (under 60 lines)
  • specs/*.md - requirement documents
  • IMPLEMENTATION_PLAN.md - persistent shared state

Critical Rules

  • One task per iteration (prevents context bloat)
  • Plans must remain under 60 lines
  • Topic scope test: describe each spec file in one sentence without conjunctions
  • Set iteration limits upfront (50+ iterations can cost $50-100+)

Backpressure Mechanisms

In priority order:

  1. Tests and build validation - deterministic gates
  2. Code patterns - guide agent behavior implicitly
  3. LLM-as-judge - for subjective criteria (tone, UX feel)
