## **MusicForge**

*389 prompts · 23 active days*

MusicForge — a collaborative music app for performing musicians — received the most sustained attention this month, with work happening nearly every single day. The scope was broad: from deep backend infrastructure to pixel-level UI polish, all in service of making the app genuinely useful for real rehearsals and gigs.

The month began with a significant data quality fix: song titles were being pulled from filenames rather than the actual LilyPond source file headers, which meant 129 songs had wrong diacritics, punctuation, or missing subtitles. A fallback system was built so that existing setlist references wouldn't break. From there, a major **setlist interchange format** (`.musicforge.json`) was designed and shipped — enabling export and import of setlists between devices and users, with key conversion utilities and 42 new tests. This was a collaboration point with Eric, the project's other core contributor, who gave feedback on the format design.

A big push went into the **songwriter editor**, a new feature that lets users write chord charts directly in the app using a custom text format called BarStock. This included stanza breaks, beat markers, chord inheritance (where a verse inherits chords from the previous one), inline validation with one-click fixes, and a preview panel with pretty-printed chords. The songwriter feeds into a **WeasyPrint PDF pipeline** — replacing the heavier LilyPond engraving system for text-only charts — with polished typography (TeX Gyre fonts), measure numbers, metadata blocks, and branded footers. The editor also got a metronome, scroll sync between the editing and preview panes, and multi-column layout.

The **metronome** itself was a multi-week effort. It started with a full 8-phase web build (engine, overlay, floating control, beat visuals, per-song tempo persistence), then went through extensive user testing and polish: removing haptic feedback (iPads don't have it), fixing beat indicator sync bugs, making controls translucent so sheet music stays readable, and adding tap-tempo with a redesigned concentric-circle icon. A subtle but critical bug surfaced late in the month: the app was defaulting to 120 BPM instead of reading the correct tempo from the catalog database — traced to a frontend issue where the Firestore value wasn't loaded in time. Five tempo-parsing macros were handled, 24 new tests added.

**Groove Sync** — the real-time collaborative feature where multiple musicians follow a leader's position in a setlist — got participant tracking, follower minimize/return, change notifications, and a song-change banner. The state machine was extracted into a testable reducer. The project also saw a **Firestore song registry** implemented as a single source of truth for song metadata, documentation consolidated from 15 to 9 files, and the test suite growing to 320+ tests. Deployments went to Fly.io (backend) and S3 (catalog) throughout the month.

## **Person-Tracking**

*899 prompts · 14 active days*

This is a computer vision research project focused on tracking skiers on mogul slopes — specifically, estimating a skier's position on a slope called the Swiss Wall using video footage, IMU sensors (motion/orientation data from wearable devices), and ground control points (known physical landmarks like poles and rocks). It's a collaboration with a researcher named Petr.

The month started with a major architectural restructuring. The system's core algorithm, called RSEP (Recursive State Estimation Process), was reframed from Kalman filter language (predict-search-update) to a **visual SLAM formulation** — essentially, the same class of problem that self-driving cars solve when mapping their environment. Six named domain estimators were defined: Camera Pose Solver, Mogul Detector, Mogul Localizer, POI Extractor, NTZB Line Estimator, and Skier Position Estimator. New documentation and a Mermaid process flow diagram were created. This was done in close coordination with Petr, incorporating his feedback on terminology and approach.

On the implementation side, work progressed on **IMU-based slope estimation** — using accelerometer and gyroscope data to figure out the slope's pitch and direction. Low-pass filtering and skiing detection were added to clean up noisy sensor data, with empirical correction factors identified to account for head-lean bias. A **mogul detector prototype** was built using LoG (Laplacian of Gaussian) blob detection, but it hit a wall: the camera pose solver produced degenerate results because the ground control points were nearly colinear (all in a line), which makes the math unsolvable. The fix required re-annotating reference points to include non-colinear landmarks.

A **PTL (Point Tracking and Localization) optimizer** was built for multi-frame detection, but testing revealed that Sobel edge detections (~97 per frame) were overwhelmingly noisy, clustering into a single blob that drowned out the actual mogul signal. A detailed email was drafted to Petr outlining the problem and asking for guidance. Petr's feedback led to important corrections: distance-map spatial filtering needs to account for IMU rotation, Hessian ridge detection should be dropped, and the database's geometric query capabilities should be leveraged. Twenty-two unit tests were added for the analysis pipeline framework, and the annotation UI got a detection overlay with parameter sliders. The branch was pushed and is ready for the next round of collaboration.

## **Home Assistant**

*214 prompts · 12 active days*

Home Assistant is a home automation platform, and this project involves building and tuning a sophisticated lighting system for a multi-floor house. The work this month was a mix of debugging frustrating real-world issues and building out a more robust architecture.

The biggest change was a **complete rewrite of the mood and presence system**. The old approach used 8 time-of-day phases (Sleep, Wake, Morning, Afternoon, Dusk, Evening, Night, etc.) and a location tracker — but it was fragile and hard to reason about. It was simplified to 4 phases (Sleep, Day, Eve, Night) and a new mood system with 7 named levels (Off, Dim, Soft, Cozy, Bright, Full, Minimal). Presence tracking was overhauled to replace location sensing, and about 25 stale entities were cleaned up from the system.

A recurring villain this month: the dog. The household dog kept triggering motion sensors on the stairs, causing the system to think someone had changed floors, which would kill the lights in the room the human was actually in. The first fix was a dedicated dog detection system, but that was overly complex. It was replaced with a simpler **2-minute cooldown** on location changes plus a **presence veto** — if motion sensors on the current floor are active, the system blocks location flips. A dead Eve sensor battery caused similar phantom flips and was tracked down.

Three new Eve Motion sensors were added (desk/piano area, laundry room, dining room), and the garage sensor was repurposed for the powder room. New automations include: laundry lights with illuminance-based brightness, a two-step powder room dimout, dining room color shifting based on occupancy (warm white when occupied, triadic colors when empty), and a desk arrival welcome. A **Flask dashboard** was built with four pages: script triggers, brightness tuning sliders, feedback trends, and an automations reference. The bathroom auto-off-during-shower problem was fixed by removing the motion timeout, and a light-off tracker with persistent notifications was added for debugging.

## **Soirée**

*214 prompts · 13 active days*

Soirée is an event invitation and RSVP management app — think a simple, self-hosted alternative to Evite. It went from bootstrapped prototype to production-deployed app over the course of the month.

The project started on February 23rd with foundational decisions: Firebase anonymous auth for guests (with an upgrade path to Google sign-in), custom claims for organizers, and a multi-event Firestore schema. Four Vercel serverless functions were scaffolded. Over the next few days, the app was deployed to production at soiree.pianohouseproject.org with full end-to-end functionality — invites send via Resend (an email API service), guests can RSVP with party size, and responses aggregate correctly. Firestore security rules were debugged and deployed.

A major **migration from Vite+React Router to Next.js 15** with shadcn/ui (a component library) was completed in late February, including updating all environment variables and redeploying. Firebase auth required significant debugging — a missing `init.json` file was the root cause of authentication failures, and the OAuth flow needed switching from popup to redirect.

Feature development moved quickly through versioned releases: v0.3 (RSVP fixes, response banners, guest sign-in), v0.5 (draft events, bulk invite with 30-guest cap, organizer comment editing), v0.6-v0.7 (name editing, nudge/thank-you emails, email preview, shareable draft links), and v0.8 (client-driven bulk sends with progress, guest chat as a multi-message bulletin board, CSV import/export). A copy-invite feature was added as a workaround for new-domain email spam issues. Rate limiting was handled with sequential sends and 600ms delays. Max headcount, email send tracking, and reminder emails rounded out the feature set.

The project also spawned **Freevite** — a public fork of Soirée rebranded across 14 files, deployed to free-vite.com with its own Firebase project, Vercel deployment, and Cloudflare DNS. A DESIGN.md was created documenting 8 long-term architecture tensions.

## **Prompt Lab**

*191 prompts · 16 active days*

Prompt Lab is a developer productivity tool — a personal dashboard and automation system that tracks work sessions across all projects, generates summaries, sends email digests, and provides an analytical layer over daily development activity. It's the system that makes reviews like this one possible.

The month's work centered on reliability and portability. **Crontab was replaced with launchd** (macOS's native task scheduler) for the todo email and synthesis jobs, after debugging why emails had silently stopped sending — the cron job had never actually fired. An `install.sh` script was created for new-machine setup, with platform-agnostic placeholders. The slash commands (custom CLI shortcuts) were restructured: `workflow/commands/` became the source of truth, synced to `~/.claude/commands/`.

The **dashboard got a project-centric redesign** using Preact (a lightweight React alternative) with per-project cards, server-side project hiding, and a new project detail page showing AI-generated session summaries, intentions, next steps, and status controls. A **Todos view** was added that scrapes all project files for Next Steps/Backlog/Planned sections and displays them grouped by project. Token usage monitoring was implemented with per-session tracking and peak usage indicators.

Code quality improvements included fixing database connection leaks, replacing an N+1 query pattern with batched queries, and simplifying the `/readup` and `/handoff` commands (cut by ~50% in line count while keeping the valuable parts). The email pipeline was upgraded to use **Anthropic tool use** for structured output — replacing fragile JSON text parsing with proper structured API calls — and switched to Sonnet 4.6 with 16K token allocation.
